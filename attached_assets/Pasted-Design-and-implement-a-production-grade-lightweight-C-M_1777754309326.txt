Design and implement a production-grade, lightweight C++ Minecraft launcher named “Genesis” based on HMCCC-style architecture as a structural reference point, but do not copy code directly; instead, replicate its separation of concerns and runtime flow while modernizing it for clarity, maintainability, and cross-platform support, targeting macOS and Windows first, with Linux compatibility as a secondary goal, and begin by defining a clean modular architecture where the launcher is split into distinct subsystems including authentication, version management, instance management, asset handling, JVM orchestration, and update/patch management, ensuring that each subsystem is fully isolated behind interfaces so that core logic does not depend on platform-specific implementations or external services directly but instead interacts through well-defined abstraction layers.

Implement a launcher core written in modern C++ (C++20 or later) that manages lifecycle state transitions such as initialization, authentication, profile resolution, instance selection, game launch preparation, JVM configuration, process spawning, and runtime monitoring, ensuring that each state transition is explicit, observable, and recoverable in case of failure, and design the system so that every operation returns structured result objects rather than raw booleans or exceptions for normal flow control, reserving exceptions only for truly unrecoverable system-level errors.

Build an authentication module that supports Microsoft OAuth-based Minecraft authentication, including token acquisition, refresh flow, and secure local persistence of credentials using OS-level secure storage APIs where available (Keychain on macOS, Credential Manager on Windows), and ensure that no raw tokens are ever stored in plaintext files, logs, or memory dumps beyond their required lifecycle scope, and include explicit token expiry detection and automatic re-authentication workflows that fail gracefully when offline or when authentication services are unavailable.

Create a version management system that can query, cache, and resolve Minecraft versions dynamically from Mojang’s official metadata endpoints, while also supporting custom modded instances, and ensure that version manifests are validated against schema definitions before use, with fallback behavior for corrupted or partially downloaded manifests, and include support for multiple runtime configurations per version such as Forge, Fabric, and vanilla profiles, treating each as a composable extension rather than separate code paths.

Implement a robust asset and library manager that handles download, verification, and caching of game assets, libraries, and native dependencies using hash validation (SHA-1 or stronger where available), parallelized download queues with retry policies, and resumable downloads for interrupted sessions, ensuring that asset integrity is always verified before launch and that corrupted assets trigger automatic repair workflows rather than silent failure.

Design a JVM orchestration layer that dynamically constructs JVM arguments based on selected version, system capabilities, memory constraints, and user-defined profiles, including intelligent defaults for heap sizing, garbage collection tuning, and native library paths, while ensuring that unsafe or invalid JVM configurations are rejected with clear diagnostic messages rather than passed through to the runtime, and ensure that process spawning is handled in a platform-abstracted way that captures stdout/stderr streams for logging and crash analysis without blocking the main UI thread.

Build an instance management system that allows multiple isolated Minecraft installations, each with independent configuration, mods, saves, and resource packs, ensuring that instances are fully sandboxed from each other at the filesystem level, and implement a clean directory structure strategy that avoids global state leakage and supports portable instances that can be moved across machines without breaking references.

Create a lightweight but responsive UI layer (Qt, ImGui, or similar depending on target choice) that exposes instance selection, version selection, authentication status, download progress, and launch controls, ensuring that UI state is always derived from backend state rather than stored independently, and that all long-running operations are asynchronous and non-blocking with clear progress reporting channels.

Implement a logging and diagnostics subsystem that captures structured logs across all modules with severity levels, timestamps, and contextual metadata, ensuring logs can be exported for debugging but never include sensitive authentication data, and include a crash handler that generates reproducible diagnostic bundles containing environment information, version state, and last known launcher state without exposing private credentials.

Add an update system for the launcher itself that supports self-updating binaries in a safe manner using versioned release channels, checksum validation, and rollback capability if an update fails or produces a non-functional binary, ensuring atomic replacement of executable files where supported.

Ensure the entire system is designed with deterministic behavior where possible, particularly in asset resolution, version selection, and instance loading order, so that repeated launches with identical configuration produce identical runtime environments, and explicitly document all external dependencies and system assumptions.

Finally, structure the codebase so that it is easy to extend with modpack ecosystems like CurseForge or Fabric mod loaders in the future without requiring rewrites of core systems, and prioritize clean interfaces, minimal global state, explicit dependency injection, and clear separation between platform logic, business logic, and external API integration, producing a launcher that is not only functional but maintainable, auditable, and suitable for long-term evolution into a full modded ecosystem manager.
